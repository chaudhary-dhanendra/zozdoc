# Architecture Decision Records

## Purpose

ADRs record significant architectural decisions for HermesNet. They preserve context, decision rationale, consequences, and supersession history.

## Naming

Use `ADR-0001-short-title.md`. Numbers are monotonically increasing and never reused.

## Lifecycle

ADRs begin as drafts, receive review, and are accepted, rejected, deprecated, or superseded. Accepted ADRs are not edited to change decisions; create a new ADR instead.

## Status Values

- `Draft`: under preparation.
- `Accepted`: approved and authoritative.
- `Rejected`: considered and not adopted.
- `Deprecated`: no longer recommended but retained for compatibility.
- `Superseded`: replaced by a later ADR.

## Writing Future ADRs

Each ADR should include title, status, date, owners, context, decision, consequences, alternatives, compatibility impact, affected volumes, and references.
